## 3. Проблема

### Какую проблему решаем

Современные LLM-системы часто используют одну модель для всех типов запросов. Это приводит к двум типовым проблемам:

1. Простые запросы обрабатываются слишком дорогими моделями.
2. Сложные запросы, наоборот, могут попадать в недостаточно мощную модель и терять в качестве.

Дополнительно возникает ещё несколько практических болей:

- нет автоматического выбора между дешёвой и дорогой моделью;
- нет маршрутизации между API-моделями, локальными моделями и агентным режимом;
- сложно учитывать приватность данных;
- нет удобного fallback между провайдерами;
- стоимость эксплуатации быстро растёт вместе с нагрузкой.

### Кто с этим сталкивается и почему это больно

С этим сталкиваются:

- команды, которые внедряют AI-функции в SaaS-продукты;
- разработчики внутренних корпоративных AI-ассистентов;
- стартапы с ограниченным бюджетом на inference;
- команды, которым нужно балансировать качество, latency, privacy и стоимость.

Почему это больно:

- растут расходы на API;
- качество ответов нестабильно;
- ручной выбор модели плохо масштабируется;
- сложно строить production-пайплайн с fallback и политиками маршрутизации.

### Как справляются сейчас

Чаще всего используют один из трёх подходов:

- отправляют все запросы в одну сильную модель и переплачивают;
- отправляют всё в дешёвую модель и получают падение качества;
- собирают вручную набор `if/else`-правил, которые быстро становятся трудно поддерживаемыми.

### Масштаб проблемы

Даже для команды со средним AI-трафиком проблема становится заметной уже на десятках тысяч запросов в месяц:

- лишние 20–50% затрат на inference;
- рост времени на поддержку и ручную настройку маршрутизации;
- увеличение риска ошибок при деградации провайдера или перегрузке модели.

## 4. Решение

### Что предлагаем

LLM Router Agent — это интеллектуальный слой маршрутизации между пользовательским запросом и набором моделей / агентов.  
Система анализирует запрос и выбирает оптимальный маршрут:

- small LLM для простых задач;
- large LLM для более сложных;
- agent с инструментами для многошаговых сценариев.

Цель проекта — одновременно снизить стоимость обработки запросов и сохранить или повысить качество ответов.

### Что делает LLM в решении

LLM здесь играет **роль ядра маршрутизации и обработки**, а не только вспомогательного компонента.

Система использует LLM в двух ролях:

1. как исполнитель запроса;
2. как роутер, который определяет, какой маршрут лучше подходит для текущего запроса.

### Почему именно LLM, а не другой подход

Потому что задача выбора маршрута зависит не только от формальных правил, но и от смысла запроса:

- насколько запрос сложный;
- нужен ли reasoning;
- достаточно ли лёгкой модели;
- нужен ли вызов инструментов.

Часть этого можно покрыть эвристиками, но для гибкой адаптации нужен LLM-aware routing.

### Технический подход

Текущая методология проекта:

- классификация запроса по сложности и типу;
- маршрутизация между:
  - small LLM,
  - large LLM,
  - agent with tools;
- использование LLM как роутера;
- добавление эвристик и правил;
- оценка качества и стоимости.

Планируемые расширения:

- RAG;
- reranker;
- обучение роутера;
- multi-agent сценарии;
- evaluation pipeline.

### Пользовательский сценарий

Пользователь задаёт вопрос.  
Система сначала анализирует тип запроса. Если запрос простой, он отправляется в дешёвую модель. Если запрос требует рассуждений или внешних инструментов, он направляется либо в более сильную модель, либо в агентный режим.  
В результате пользователь получает ответ нужного качества, а система не тратит дорогой inference там, где он не нужен.

### Ожидаемые результаты

- снижение стоимости генерации до 30–70% в зависимости от профиля нагрузки;
- ускорение ответа на простых запросах;
- сохранение или улучшение качества;
- демонстрация работы системы на разных типах запросов.

## 5. Деньги

Самый важный раздел для этого проекта, потому что LLM Router Agent создаёт ценность не только качеством, но и управлением расходами на inference.

### 5.1. Основные допущения для расчётов

Ниже грубая, но практичная оценка для MVP / пилота.  
Порядок величин важнее абсолютной точности.

Базовые допущения:

- система обрабатывает от 50 000 до 150 000 запросов в месяц;
- используется гибридная схема:
  - часть запросов идёт в дешёвые API-модели;
  - часть в более сильные модели;
  - часть может идти в локальную модель или через OpenRouter;
- в MVP ещё нет тяжёлой enterprise-инфраструктуры, но уже есть расходы на hosting, логи, мониторинг и тестовый inference.

### 5.2. Сколько стоит сделать и эксплуатировать

#### Сводная таблица

| Раздел | Сумма | Комментарий |
|---|---:|---|
| Разработка / настройка (разово) | $4 000 – $8 000 | MVP: backend, routing logic, интеграции, eval, деплой |
| Инфраструктура и API (в месяц) | $250 – $900 | Хостинг + API-модели + OpenRouter + тестовый inference |
| Поддержка (в месяц) | $300 – $1 000 | 10–30 часов инженера на поддержку, эксперименты и обновления |

#### Что входит в разовую разработку

- проектирование архитектуры роутера;
- интеграции с LLM-провайдерами;
- routing logic;
- правила и fallback;
- логирование и трассировка маршрутов;
- базовая система оценки качества и стоимости;
- подготовка демо / MVP.

#### Что входит в ежемесячную инфраструктуру и API

1. **Базовая инфраструктура**

- приложение / backend;
- небольшой сервер или контейнер;
- логирование;
- мониторинг;
- хранение конфигов и секретов.

Ориентир:

- `$20 – $80 / месяц`

2. **API-модели**

Если использовать OpenAI / Anthropic / OpenRouter для части трафика:

- пилот с умеренным трафиком: `$100 – $500 / месяц`
- при более активном использовании сильных моделей: `$500+ / месяц`

3. **OpenRouter**

Если OpenRouter используется как основной мульти-провайдерный слой:

- к стоимости самих моделей добавляется платформа / комиссия;
- это удобно для гибкости и fallback, но немного увеличивает стоимость по сравнению с прямым вызовом API.

4. **Локальные модели**

Если часть маршрутов идёт в локальный inference:

- аренда GPU или поддержка собственного GPU-сервера становится отдельной статьёй расходов.

Ориентир по облачной GPU-аренде для локальных моделей:

- A10 / 24 GB, неполная загрузка: примерно `$150 – $250 / месяц`
- A10 / A6000 / 24/7: примерно `$600 – $700 / месяц`
- A100 / 24/7: от `$1 000 / месяц`

Вывод:

- для MVP выгоднее начинать с API-only или hybrid;
- локальный контур экономически оправдан, если важны приватность, контроль и достаточно высокий объём простых запросов.

### 5.3. Учёт разных сценариев инфраструктуры

#### Сценарий A. Только API-модели

Подходит для MVP и раннего пилота.

Плюсы:

- быстрее запуск;
- почти нет DevOps-сложности;
- не нужно держать GPU.

Минусы:

- переменная стоимость растёт вместе с трафиком;
- хуже контроль над приватностью;
- меньше гибкости при vendor lock-in.

Оценка:

- `$120 – $500 / месяц` при небольшом и среднем пилотном трафике.

#### Сценарий B. API + OpenRouter + локальная модель

Это наиболее реалистичный сценарий для проекта.

Плюсы:

- дешёвые запросы можно уводить в локальную модель;
- сложные — в сильные API;
- OpenRouter удобно использовать как единый слой доступа к нескольким моделям;
- проще делать fallback.

Минусы:

- выше сложность поддержки;
- появляется необходимость считать и API-стоимость, и GPU-стоимость.

Оценка:

- `$250 – $900 / месяц` для MVP / пилота;
- при постоянном локальном GPU-контуре расходы могут вырасти до `$700 – $1 600 / месяц`.

#### Сценарий C. Упор на локальные модели

Подходит для on-prem / privacy-sensitive сценариев.

Плюсы:

- выше контроль над данными;
- предсказуемая фиксированная часть затрат;
- меньше зависимости от внешних API.

Минусы:

- нужны GPU и поддержка;
- может просесть качество на сложных задачах;
- выше порог входа по инфраструктуре.

Оценка:

- `$600 – $1 500+ / месяц` в зависимости от выбранной GPU-конфигурации и режима работы.

### 5.4. Какой эффект в деньгах

Основной экономический эффект проекта — это **снижение средней стоимости обработки одного запроса** без критического падения качества.

#### Таблица эффекта

| Источник эффекта | Как считаем | Сумма в месяц |
|---|---|---:|
| Снижение затрат на inference | 25–40% экономии относительно схемы “всё отправляем в сильную модель” | $300 – $1 500 |
| Экономия времени инженеров | 10–20 часов в месяц меньше на ручную маршрутизацию, дебаг и переключение провайдеров | $200 – $600 |
| Снижение риска простоя | fallback между маршрутами уменьшает потери при сбоях провайдера | сложно оценить, но эффект может быть существенным |
| Более дешёвый масштаб | рост нагрузки не линейно ведёт к росту расходов, если часть трафика уходит в лёгкие маршруты | зависит от трафика |

### 5.5. Пример грубого расчёта

Предположим, без роутера все запросы идут в сильную модель, и стоимость inference составляет:

- `$2 000 / месяц`

После внедрения роутера:

- 60% запросов уходят в дешёвые маршруты;
- 30% — в средние;
- 10% — в дорогие / агентные.

Тогда общая стоимость может снизиться до:

- `$1 100 – $1 400 / месяц`

Экономия:

- `$600 – $900 / месяц`

Если при этом на поддержку тратится ещё `$300 – $500 / месяц`, чистый эффект всё равно остаётся положительным:

- примерно `$300 – $600 / месяц`

### 5.6. Когда окупится

Если считать проект как внутренний инструмент или B2B-прототип:

- при хорошем трафике и реальной экономии на inference окупаемость возможна через **4–9 месяцев**;
- при низком трафике проект может долго не окупаться напрямую, но оставаться стратегически полезным как инфраструктурный слой.

### 5.7. Что будет, если не делать проект

- расходы на LLM будут выше, чем могли бы быть;
- часть запросов будет обслуживаться неоптимальными моделями;
- команда останется без единой политики маршрутизации;
- сложнее будет добавлять новые модели и провайдеров;
- сильнее зависимость от одного API-провайдера;
- при росте трафика стоимость будет масштабироваться хуже.

### 5.8. Вывод по разделу “Деньги”

Для LLM Router Agent деньги — это не побочная тема, а одна из главных причин существования проекта.

Ключевая бизнес-гипотеза:

> проект снижает среднюю стоимость обработки пользовательского запроса и даёт более управляемую AI-инфраструктуру за счёт интеллектуальной маршрутизации между дешёвыми, дорогими, локальными и агентными маршрутами.

Самый реалистичный путь для MVP:

- сначала API-only или hybrid,
- затем добавление локальных моделей для privacy / cost-sensitive сценариев,
- затем развитие в open-core, B2B hosted или on-prem продукт.

## 7. Безопасность и ограничения

### Главные риски

1. **Неправильная маршрутизация**

Система может отправить сложный запрос в слишком слабую модель.  
Как снижаем риск:

- правила и эвристики поверх LLM-router;
- fallback на более сильную модель;
- evaluation pipeline.

2. **Слишком сложная эксплуатация**

При добавлении большого количества провайдеров проект может стать сложным в поддержке.  
Как снижаем риск:

- единый интерфейс интеграции;
- централизованная конфигурация маршрутов;
- логирование и трассировка решений роутера.

3. **Рост инфраструктурной стоимости**

Локальные модели могут увеличить fixed cost.  
Как снижаем риск:

- использовать локальные модели только там, где это оправдано;
- считать cost per request по каждому маршруту;
- начинать с MVP без избыточной GPU-инфраструктуры.

### Есть ли работа с персональными данными

Потенциально да, если проект используется в продуктах с пользовательскими запросами.  
Поэтому важно предусмотреть:

- разделение приватных и непубличных запросов;
- возможность локальной обработки;
- masking / redaction чувствительных данных;
- контроль логирования.

### Как обеспечивается безопасность

- маршрутизация приватных запросов в локальный или доверенный контур;
- ограничение логирования чувствительных данных;
- централизованное хранение ключей и конфигураций;
- возможность отключать внешние API для чувствительных сценариев.
